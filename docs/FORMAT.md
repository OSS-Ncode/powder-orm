# PCB — Powder Columnar Buffer (v1)

PCB는 Powder가 쿼리 결과를 Rust 코어에서 호스트 언어로 옮길 때 쓰는 on-the-wire, zero-copy 컬럼 포맷이다. 의도적으로 Arrow와 같은 모양(validity 비트맵 + 타입별 데이터 버퍼)이지만 범위가 훨씬 작아서, 어떤 언어에서든 리더를 ~120줄로 작성할 수 있다.

하나의 쿼리 결과는 **하나의 연속된 바이트 버퍼**다. 모든 정수는 **리틀 엔디언**이다. 모든 데이터 버퍼는 **8바이트 경계**에서 시작하며, 이 덕분에 리더가 숫자 버퍼를 `Float64Array` / `BigInt64Array`(Node)나 `memoryview.cast('d'/'q')`(Python)로 **값 단위 복사 없이** 앨리어스할 수 있다.

## 전체 레이아웃

```
┌────────────┬──────────────────┬───────────────┬──────────────────────────┐
│  헤더      │   디렉터리       │   이름        │   데이터 버퍼            │
│  24바이트  │  40 × ncols      │   (utf-8)     │   (각각 8바이트 정렬)     │
└────────────┴──────────────────┴───────────────┴──────────────────────────┘
```

## 헤더 (24바이트)

| 오프셋 | 크기 | 필드               | 비고                      |
| ------ | ---- | ------------------ | ------------------------- |
| 0      | 4    | `magic`            | ASCII `PCB1`              |
| 4      | 2    | `version`          | `1`                       |
| 6      | 2    | `flags`            | 예약, `0`                 |
| 8      | 4    | `num_columns`      |                           |
| 12     | 4    | `num_rows`         |                           |
| 16     | 4    | `directory_offset` | 절대 오프셋; 현재 `24`    |
| 20     | 4    | `reserved`         | `0`                       |

## 디렉터리 엔트리 (각 40바이트, `num_columns`개)

| 오프셋 | 크기 | 필드           | 비고                                         |
| ------ | ---- | -------------- | -------------------------------------------- |
| 0      | 4    | `name_off`     | 컬럼 이름의 절대 오프셋                      |
| 4      | 4    | `name_len`     | UTF-8 이름의 바이트 길이                     |
| 8      | 1    | `dtype`        | `0`=Int64 `1`=Float64 `2`=Bool `3`=Utf8      |
| 9      | 1    | `flags`        | bit0 = validity 비트맵 존재; bit1 = Utf8 문자 데이터가 순수 ASCII (힌트: 리더가 버퍼 전체를 한 번에 디코드하고 슬라이스해도 됨; 무시해도 항상 올바름) |
| 10     | 2    | `reserved`     |                                              |
| 12     | 4    | `validity_off` | 절대 오프셋; bit0이 설정된 경우에만 유효     |
| 16     | 4    | `validity_len` | 바이트 수 = `ceil(num_rows / 8)`             |
| 20     | 4    | `buf1_off`     | 기본 데이터 버퍼 (아래 참고)                 |
| 24     | 4    | `buf1_len`     |                                              |
| 28     | 4    | `buf2_off`     | Utf8 문자 데이터; 그 외 타입은 `0`           |
| 32     | 4    | `buf2_len`     |                                              |
| 36     | 4    | `reserved`     |                                              |

## 타입별 버퍼

| 타입    | `buf1`                                       | `buf2`            |
| ------- | -------------------------------------------- | ----------------- |
| Int64   | `num_rows × i64` (LE)                         | —                 |
| Float64 | `num_rows × f64` (LE)                         | —                 |
| Bool    | 비트 팩, LSB 우선, `ceil(n/8)` 바이트         | —                 |
| Utf8    | `(num_rows + 1) × u32` 오프셋 (LE)            | UTF-8 문자 데이터 |

## Validity 비트맵

LSB 우선: 행 `r`의 값은 `bitmap[r >> 3] & (1 << (r & 7))`일 때 유효하다. `has validity` 플래그가 꺼져 있으면 모든 값이 유효하다 (흔한 non-null 케이스는 바이트를 전혀 쓰지 않는다).

## Zero-copy로 읽기

`buf1_off`가 8바이트 정렬이고 오프셋이 절대값이므로, 리더는 전송된 바이트 위에 뷰를 바로 만든다:

- **Node** — `new Float64Array(buf.buffer, buf.byteOffset + buf1_off, num_rows)`
- **Python** — `memoryview(buf)[buf1_off : buf1_off + num_rows*8].cast('d')`

둘 다 엔진 출력을 앨리어스한다; 요소 단위 복사가 일어나지 않는다. 문자열과 null 체크 접근은 여전히 오프셋/validity 버퍼를 거치지만, 이들 역시 같은 버퍼의 슬라이스다.
